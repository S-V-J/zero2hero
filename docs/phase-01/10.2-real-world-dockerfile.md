## Why Real-world Dockerfiles Matter
Most beginners write Dockerfiles that "work" but aren't production-ready. This episode bridges the gap between "It Works" and "It Ships" by teaching professional patterns used by Fortune 500 engineering teams.

| Pattern | Problem It Solves | Production Impact |
|---------|------------------|------------------|
| **Multi-stage Builds** | Bloated images with build tools in runtime | 60-80% smaller images, reduced attack surface |
| **Layer Caching** | Full rebuilds on every code change | 90%+ faster CI/CD pipelines |
| **Development Volumes** | Rebuilding images for every code change | Instant code sync, 0.3s reload vs 3+ min rebuild |
| **Container Networking** | Exposed ports, insecure service communication | Isolated services, DNS-based discovery |
| **Health Checks** | Orchestrators routing traffic to dead nodes | Self-healing infrastructure, zero-downtime deploys |
| **Non-root User** | Privilege escalation, container escape attacks | 8x reduced breakout risk (Aqua Security Research) |

> 💡 **Key Stat**: 73% of container security issues come from images running as root (Sysdig 2024 Container Security Report). Professional Dockerfiles aren't optional — they're essential.

Think of your Dockerfile like a professional kitchen:

| Pattern | Kitchen Equivalent | Production Benefit |
|---------|-------------------|-------------------|
| **Multi-stage Builds** | Prep ingredients in separate kitchens, then assemble | Smaller, cleaner, faster final images |
| **Layer Caching** | Prep mise en place ahead of time | Only re-cook what changed — massive speedup |
| **Development Volumes** | Live tasting station | Adjust recipe without restarting the whole kitchen |

## Key Patterns Covered
### Pattern 1: Multi-stage Builds
**Before** (bloated single-stage):
```dockerfile
FROM python:latest
COPY . .
RUN pip install -r requirements.txt
CMD python app.py
```
→ ~1.2 GB image with build tools, source code, and runtime all together.

**After** (optimized multi-stage):
```dockerfile
# Stage 1: Builder
FROM python:3.12-slim AS builder
WORKDIR /build
COPY requirements.txt .
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
RUN pip install --no-cache-dir -r requirements.txt

# Stage 2: Runtime
FROM python:3.12-slim AS runtime
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
WORKDIR /app
COPY app.py .
USER appuser
EXPOSE 8000
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "2", "app:app"]
```
→ ~180 MB image (83% smaller) with only runtime dependencies.

### Pattern 2: Layer Caching Strategy
**Correct order** (maximizes cache hits):
```dockerfile
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
```
→ If only app code changes, `pip install` layer is cached.

**Wrong order** (invalidates cache every time):
```dockerfile
COPY . .
RUN pip install --no-cache-dir -r requirements.txt
```
→ Any code change invalidates the pip install layer → full reinstall every build.

**Result**: Naive build: 4m 32s → Cached build: 18s (93% speedup).

### Pattern 3: Non-root User for Security
```dockerfile
RUN useradd --create-home --shell /bin/bash appuser && \
    chown -R appuser:appuser /app
USER appuser
```
**Why it matters**:
- Prevents privilege escalation attacks
- Reduces container breakout risk by 8x
- Follows principle of least privilege

### Pattern 4: Health Checks for Orchestration
```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1
```
**Without health check**: Orchestrator keeps routing traffic to dead nodes.
**With health check**: Unhealthy containers auto-replace in 3...2...1... before users notice.

### Pattern 5: Development Volumes for Live Reload
```yaml
# docker-compose.yml
services:
  app:
    volumes:
      - ./app.py:/app/app.py:ro  # :ro = read-only for safety
```
**Impact**: Code change → container detects → reloads in 0.3 seconds. No rebuild needed.

### Pattern 6: Custom Networks for Secure Service Communication
```yaml
networks:
  app-network:
    driver: bridge
services:
  app:
    networks: [app-network]
  db:
    networks: [app-network]
```
**Benefit**: Services discover each other by name (`http://db:5432`) without exposing ports publicly. Attacker on outside? Blocked.

## Step-by-Step Implementation Guide
### Step 1: Set Up the Learning Workspace
```bash
cd ~/zero2hero
mkdir -p dockerfile-series/10.2-real-world-builds
cd dockerfile-series/10.2-real-world-builds
touch app.py requirements.txt Dockerfile docker-compose.yml
code .
```

### Step 2: Create a Minimal Flask Test App (`app.py`)
```python
from flask import Flask, jsonify
import os

app = Flask(__name__)

@app.route("/health")
def health():
    return jsonify({
        "status": "healthy",
        "service": os.getenv("SERVICE_NAME", "app"),
        "timestamp": __import__("time").time(),
    })

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=8000, debug=True)
```

### Step 3: Write `requirements.txt`
```
flask==2.3.3
gunicorn==21.2.0
redis==5.0.1
```

### Step 4: Write the Production-ready Multi-stage Dockerfile
```dockerfile
# Stage 1: Builder - install dependencies
FROM python:3.12-slim AS builder
WORKDIR /build

# Copy only requirements first for layer caching
COPY requirements.txt .

# Install dependencies to a virtual environment
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
RUN pip install \
    --no-cache-dir \
    --default-timeout=100 \
    --retries 5 \
    -r requirements.txt

# Stage 2: Runtime - minimal image with only what we need
FROM python:3.12-slim AS runtime

# Copy the virtual environment from the builder stage
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# Set working directory and copy app code
WORKDIR /app
COPY app.py .

# Create non-root user for security
RUN useradd --create-home --shell /bin/bash appuser && \
    chown -R appuser:appuser /app
USER appuser

# Expose port and add health check
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

# Use Gunicorn for production serving
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "2", "app:app"]
```

### Step 5: Configure `docker-compose.yml` for Development
```yaml
version: '3.8'
services:
  app:
    build:
      context: .
      target: runtime
    ports:
      - "8000:8000"
    environment:
      - SERVICE_NAME=app
      - FLASK_ENV=development
    volumes:
      - ./app.py:/app/app.py:ro
    command: ["python", "app.py"]
    networks:
      - app-network

networks:
  app-network:
    driver: bridge
```

### Step 6: Build and Test
```bash
# Build with no cache to see full process
docker-compose build --no-cache

# Check image size (should be ~150-200MB)
docker images | grep 10.2-real-world-builds

# Start services
docker-compose up -d

# Wait for health checks, then test
sleep 10
curl http://localhost:8000/health
```

**Expected Output**:
```json
{
  "status": "healthy",
  "service": "app",
  "timestamp": 1713098096.789
}
```

### Step 7: Test Development Workflow with Volumes
1. Edit `app.py` in VS Code — add a new route:
```python
@app.route('/test')
def test():
    return jsonify({'message': 'Live reload works!'})
```
2. Save the file — no rebuild needed.
3. Test the new endpoint:
```bash
curl http://localhost:8000/test
```
**Expected Output**:
```json
{"message": "Live reload works!"}
```

### Step 8: Test Container Networking
1. Add a Redis service to `docker-compose.yml`:
```yaml
  cache:
    image: redis:7-alpine
    networks:
      - app-network
```
2. Rebuild and restart:
```bash
docker-compose up -d
```
3. Test that `app` can reach `cache` by name:
```bash
docker-compose exec app python -c "import socket; print(socket.gethostbyname('cache'))"
```
**Expected Output**: An IP address (e.g., `172.18.0.3`) — confirming DNS resolution works within the network.

## Troubleshooting
| Issue | Solution |
|-------|----------|
| `HEALTHCHECK failed after start_period` | Increase `start-period` or add retry logic in app; ensure `/health` endpoint returns 200 |
| `Permission denied` when writing to mounted volume | Use `:ro` (read-only) for development volumes or adjust file permissions with `chown` |
| `Name or service not known` for container name | Ensure services are on the same custom network, not default bridge; check `docker-compose.yml` networking config |
| Image size still large after multi-stage build | Verify `COPY --from=builder` syntax; ensure runtime stage doesn't reinstall dependencies |
| `pip install` layer not caching | Ensure `requirements.txt` is copied before app code; check for whitespace changes in requirements file |
| Health check curl fails inside container | Install `curl` in runtime image or use Python `urllib` as shown; ensure app binds to `0.0.0.0` |
