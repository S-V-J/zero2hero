## Why Write Dockerfile from Scratch?
| Concept | Purpose | Why It Matters |
|---------|---------|---------------|
| **FROM** | Selects the base image for your container | Determines OS, language runtime, security baseline, and image size |
| **WORKDIR** | Sets the working directory inside the container | Creates consistent paths; avoids permission issues; improves readability |
| **COPY vs ADD** | Copies files from host to container | COPY is explicit and preferred; ADD has extra features (URL, tar extraction) |
| **RUN** | Executes commands during image build | Installs dependencies, compiles code, prepares the environment |
| **EXPOSE** | Documents which ports the container listens on | Helps with documentation and docker-compose networking (does not publish ports) |
| **CMD vs ENTRYPOINT** | Defines the command that runs when container starts | CMD is overrideable; ENTRYPOINT is fixed; understanding both enables flexible deployment |
| **.dockerignore** | Excludes files from the build context | Reduces build time, image size, and prevents leaking secrets |
> 💡 **Chef Analogy**: Writing a Dockerfile is like creating a recipe from scratch. `FROM` is choosing your kitchen (base OS). `WORKDIR` is setting up your prep station. `COPY` is gathering ingredients. `RUN` is cooking steps. `CMD` is plating the final dish. When you write it yourself, you control every flavor, every technique, every outcome.

## Dockerfile Instructions Deep Dive
### Core Instructions (Used in Every Dockerfile)
| Instruction | Syntax Example | Purpose | Best Practice |
|-------------|---------------|---------|---------------|
| `FROM` | `FROM python:3.12-slim` | Sets the base image | Use specific tags (`3.12-slim`), not `latest`; prefer `-slim` or `-alpine` for smaller images |
| `WORKDIR` | `WORKDIR /app` | Sets working directory | Use absolute paths; create logical structure (`/app`, `/src`) |
| `COPY` | `COPY requirements.txt .` | Copies files from host to container | Copy dependency files first to leverage layer caching |
| `RUN` | `RUN pip install --no-cache-dir -r requirements.txt` | Executes build-time commands | Chain commands with `&&`; clean up caches to reduce image size |
| `EXPOSE` | `EXPOSE 8000` | Documents listening port | Does not publish port; use `-p` in `docker run` or `ports:` in compose |
| `CMD` | `CMD ["python", "app.py"]` | Sets default runtime command | Use exec form `["cmd", "arg"]` for proper signal handling |
### Advanced Instructions (For Production)
| Instruction | Syntax Example | Purpose | Best Practice |
|-------------|---------------|---------|---------------|
| `ENV` | `ENV PYTHONUNBUFFERED=1` | Sets environment variables in image | Use for app config; avoid secrets (use runtime env vars instead) |
| `ARG` | `ARG BUILD_VERSION=1.0` | Build-time variables (not in final image) | Use for versioning, feature flags during build |
| `USER` | `USER nobody` | Switches to non-root user for runtime | Improves security; run as root only for build steps |
| `HEALTHCHECK` | `HEALTHCHECK CMD curl -f http://localhost:8000/ || exit 1` | Defines container health probe | Enables orchestration tools to detect unhealthy containers |
| `LABEL` | `LABEL maintainer="you@example.com"` | Adds metadata to image | Helps with image management, scanning, and documentation |

## Step-by-Step: Writing a Python/Flask Dockerfile
### Step 1: Start with a Minimal Base Image
```dockerfile
FROM python:3.12-slim
```
**Why `python:3.12-slim`?**
- `3.12`: Specific Python version ensures reproducibility
- `-slim`: Debian-based but without extra packages → smaller image (~50MB vs ~900MB for full)
- Avoid `latest`: Prevents unexpected breaking changes when base image updates
> 💡 Pro Tip: For even smaller images, consider `python:3.12-alpine`, but be aware of `musl libc` compatibility issues with some Python wheels.
### Step 2: Set Working Directory
```dockerfile
WORKDIR /app
```
**Why set WORKDIR?**
- All subsequent `COPY`, `RUN`, `CMD` commands use this as the base path
- Creates a clean, predictable structure inside the container
- Avoids permission issues from running in `/` or `/root`
### Step 3: Copy Dependencies First (Layer Caching Optimization)
```dockerfile
COPY requirements.txt .
```
**Why copy `requirements.txt` before application code?**
- Docker caches each layer; if `requirements.txt` doesn't change, `pip install` is skipped on rebuild
- Application code changes frequently; dependencies change less often
- This pattern can reduce rebuild time from minutes to seconds
### Step 4: Install Dependencies Efficiently
```dockerfile
RUN pip install --no-cache-dir -r requirements.txt
```
**Why `--no-cache-dir`?**
- Prevents pip from storing downloaded packages in `/root/.cache`
- Reduces final image size by 50-200MB depending on dependencies
- Safe because we don't need the cache after installation
> 💡 Advanced: For multi-stage builds, install build dependencies in a builder stage, then copy only runtime dependencies to the final image.
### Step 5: Copy Application Code
```dockerfile
COPY . .
```
**What gets copied?**
- All files in the build context (current directory) except those in `.dockerignore`
- Ensure `.dockerignore` excludes: `__pycache__/`, `*.pyc`, `.git/`, `.env`, `venv/`
> ⚠️ Security Note: Never copy `.env` files or secrets into the image. Use runtime environment variables or secret managers.
### Step 6: Document the Exposed Port
```dockerfile
EXPOSE 8000
```
**What EXPOSE does (and doesn't do):**
- ✅ Documents that the container listens on port 8000
- ✅ Helps `docker-compose` and orchestration tools understand networking
- ❌ Does NOT publish the port to the host (use `-p 8000:8000` or `ports:` in compose)
### Step 7: Define the Runtime Command
```dockerfile
CMD ["python", "app.py"]
```
**Why exec form `["cmd", "arg"]`?**
- Proper signal handling: `SIGTERM` reaches your Python process for graceful shutdown
- Shell form `CMD python app.py` runs via `/bin/sh -c`, which swallows signals
- Required for production deployments with orchestration (Kubernetes, ECS, etc.)
### Step 8: (Optional) Add .dockerignore
Create a `.dockerignore` file in your project root:
```text
# Python
__pycache__/
*.py[cod]
*$py.class
*.so
.Python
venv/
.venv/
env/
.env
*.env
*.env.local

# Git
.git/
.gitignore

# IDE
.vscode/
.idea/
*.swp
*.swo

# Docker
Dockerfile
docker-compose*.yml
!.dockerignore

# OS
.DS_Store
Thumbs.db
```
**Why .dockerignore matters:**
- Excludes unnecessary files from the build context → faster builds
- Prevents leaking local secrets (`.env`) into the image
- Reduces image size by avoiding copying build artifacts

## Verification
After writing your Dockerfile, verify it works correctly:
```bash
# 1. Check Dockerfile syntax (optional)
docker build --no-cache -t test-app:dev .
# 2. Run the container locally
docker run -p 8000:8000 --rm test-app:dev
# 3. Test the endpoint in another terminal
curl http://localhost:8000
# Expected: JSON response from your Flask app
# 4. Check image size (should be ~150-250MB for slim Python + Flask)
docker images test-app:dev
# 5. Inspect image layers for optimization
docker history test-app:dev
```

## Troubleshooting
| Issue | Solution |
|-------|----------|
| `pip install` fails with permission error | Add `--user` flag or use a non-root USER in Dockerfile |
| `ModuleNotFoundError` at runtime | Ensure `requirements.txt` includes all dependencies; check `COPY . .` order |
| Image size too large (>500MB) | Use `-slim` or `-alpine` base; add `--no-cache-dir` to pip; use multi-stage builds |
| Signals not handled gracefully | Use exec form `CMD ["python", "app.py"]`, not shell form |
| Build context too slow | Add comprehensive `.dockerignore`; avoid copying large data files |
| Port not accessible from host | Use `-p 8000:8000` with `docker run` or `ports:` in `docker-compose.yml` |
### Quick Debug Commands
```bash
# Build with verbose output to see layer caching
docker build --progress=plain -t myapp:dev .
# Run container with interactive shell for debugging
docker run -it --rm myapp:dev bash
# Check environment variables inside container
docker run --rm myapp:dev env | grep MY_VAR
# Test health check manually
docker run --rm myapp:dev curl -f http://localhost:8000/ || echo "Health check failed"
```

```dockerfile
# C++ Multi-Stage Build (Episode 10.2 preview)
# Builder stage
FROM gcc:13 AS builder
WORKDIR /build
COPY . .
RUN make -j$(nproc)
# Runtime stage
FROM debian:bookworm-slim
WORKDIR /app
COPY --from=builder /build/bin/myapp .
EXPOSE 8080
CMD ["./myapp"]
```
